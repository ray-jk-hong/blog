软件层次

以下以arm-smmu-v3为例，先大致看一下软件层次。软件层次大概分为两个部分，一个是IOMMU，另一个是ARM SMMU部分。

IOMMU是Linux给各个不同的芯片和架构提供的通用的架构。

ARM SMMU部分就是ARM架构为了实现SMMU功能提供的架构。

这两个软件层次，这里先不解释。先往下看内容，看完一步再回来看一下为什么这么设计会好一点。

初始化

初始化主要分两个部分，分别对应上述两部分DTS：
第一部分是arm-smmu-v3.c读取compatible = "arm,smmu-v3"部分DTS的初始化流程。该初始化流程创建黄色的结构体。
第二部分是驱动程序读区compatible = "my_driver";的初始化部分，该初始化流程创建蓝色的结构体。

这些部分的创建，顺着代码看都能看到，所以流程不再赘述。但有些结构体或者结构体成员需要额外说明一下。

iommu_group

共享相同IO地址空间的设备的集合（一个或多个）

iommu_domain

范围和iommu_group一样，但它定义的是group范围内对应的操作的集合。每个iommu_group可以两个iommu_domain (一个default，一个非default)。

iommu_domain可以有IOMMU_DOMAIN_IDENTITY, IOMMU_DOMAIN_DMA, IOMMU_DOMAIN_UNMANAGED, IOMMU_DOMAIN_BLOCKED等类型。
IOMMU_DOMAIN_IDENTITY：iova和pa一样的场景下
IOMMU_DOMAIN_DMA：dma_xx接口处理的
IOMMU_DOMAIN_UNMANAGED：不使用dma_xx接口，直接使用iommu_map等接口处理的时候就使用这种。
`IOMMU_DOMAIN_BLOCKED：这种没见过使用…

iommu_domain的使用场景与S1/S2的使能与否有关系：
(1) SVA特性，不能在只支持S2 only的场景下工作
(2) S1/S2都支持的场景下，iommu_domain类型和应用场景对应关系如下

Use case IOMMU Domain Which Stage to be used
DMA-IOMMU mapping IOMMU_DOMAIN_IDENTITY Stage 1 bypass and stage 2 bypass
DMA-IOMMU mapping IOMMU_DOMAIN_DMA Stage 1
VFIO with VFIO_TYPE1_IOMMU or VFIO_TYPE1v2_IOMMU IOMMU_DOMAIN_UNMANAGED Stage 1
VFIO with VFIO_TYPE1_NESTING_IOMMU IOMMU_DOMAIN_UNMANAGED Stage 2
SVA IOMMU_DOMAIN_DMA Stage 1

页表

查找STE页表

第一层页表是通过StreamId查找的STE页表。StreamId的范围是从0到1 << SMMU_IDR1.SIDSIZE。
StreamId的范围还需要设置到SMMU_STRTAB_BASE_CFG.LOG2SIZE寄存器中。以arm_smmu_init_strtab_2lvl为例：

之所以要先确定StreamId的个数是因为，接下来就要根据StreamId的个数和STE页表的级数去申请内存并将内存基地址设置给STRTAB_BASE寄存器中。

上面说到的STE的页表的级数类型有两种。一种是线性的STE页表，另外一种是2级的STE列表。1级还是2级的STE排布，是根据SMMU_IDR0.ST_LEVLE寄存器的值决定的。

线性STE页表：

线性的STE排布就比较简单，就是在初始化的时候根据1 << SMMU_IDR1.SIDSIZE的值乘以64Byte算出大小并申请内存之后保存。（后续会统一设置到STRTAB_BASE中）

2级的STE列表：

2级的STE列表，是把StreamId分成两个部分。StreamId[max : split]和StreamId[split : 0]。
其中split的值定义在SMMU_STRTAB_BASE_CFG.SPLIT寄存器中，max就是StreamId的最大值和，前面已经说了是定义在1 << SMMU_IDR1.SIDSIZE。

StreamId[max : split]被用来找到Level 1 Stream Table Descriptor（L1STD），然后以L1STD中的地址为基地址，根据StreamId[split : 0]的值做偏移找到STE页表。
二级页表中，split是定义在SMMU_STRTAB_BASE_CFG.SPLIT寄存器中。

Stage-1方式

在Stage-1方式中，STE需要根据SubStreamId再找到CD（Context Descriptor）页表。
和上面查找STE方式类似，查找CD的方式也有两种。1级的和2级的。
具体支持哪种类型的，会被设置到SMMU_IDR0.CD2L中。软件在读到这个值之后，需要在STE.S1Fmt中做对应的设置。因为SMMU在后续的地址翻译过程中都是硬件行为，硬件在找到STE页表之后要决定是以哪种方式继续查找下一级CD页表。

1级的CD查找：

1级的查找比较简单，就是以STE中的S1ContextPtr基地址，根据SubStreamId做偏移找到CD即可。

2级的CD查找：

之所以搞这种多级的，是因为这种方式会在不需要大块的连续物理内存的情况下，可以支持很多个StreamId和SubStreamId。

L1CD的查找，和通过StreamId查找STE比较像，就是通过SubStreamId的高位([SMMU_IDR1.SSIDSIZE : CTXDESC_SPLIT]，不准确，但就那个意思)，再以STE.S1ContextPtr为基地址做偏移找到的。

整个CD的多少（不管是1级的还是2级的）需要保存在STE.S1CDMax中。CD的个数是2^(STE.S1CDMax)。

在STE既使能了Stage-1，也使能了Stage-2的时候，是按照如下流程翻译地址的：
1)先做Stage-1地址翻译，通过找到CD，并翻译出来地址。此时的地址就是IPA地址
2)再做Stage-2地址翻译，通过IPA地址在S2TTB指向的页表做地址翻译，找到最终的PA。

地址大小

SMMU有三个地址大小需要考虑。
1.系统输入的地址大小，可以是iova地址，可以是sva地址。
2.IAS:Intermediate Address Size
IAS影响最大的IPA地址（Stage-1翻译输出给Stage-2）。
3.OAS:Output Address Size
OAS影响最大的PA。所以定OAS的值必须要查看物理地址最大值。

CPU与SMMU交互

Command Queue（CMDQ）

Command Queue用于软件与SMMU的硬件交互，软件写命令到Command Queue，SMMU从Command Queue中读取命令处理，就是CPU写SMMU读。

(1)根据SMMU_IDR1.CMDQS指定的深度申请内存，配置基地址给SMMU_CMDQ_BASE（SMMU_CMDQ_BASE只有在 SMMU_CR0.CMDQEN为0时才能配置）。
(2)配置头尾指针寄存器SMMU_CMDQ_CONS.RD和SMMU_CMDQ_PROD.WR的初始值。
(3)配置SMMU_CR0.CMDQEN为1，使能CMDQ。

Event Queue

Event Queue用于SMMU发生软件配置错误的状态信息记录，SMMU将配置错误信息写入到Event Queue中，软件通过读取Event Queue获得配置错误信息并进行配置错误处理。SMMU写CPU读。Event Queue的最大深度由SMMU_IDR1.EVENTQS指定。

PRI Queue

SMMU_CR0.PRIQEN的值是1的时候，SMMU支持PRIQ（Page Request Queue）功能。
(1)SMMU接收PCIe RC发送的PPR（PRI page request），将PPR以PRIQ的格式写入memory，并更新PROD寄存器。
(2)软件读取PRIQ，更新指针。建立好页表后，向CMDQ写入CMD_PRI_RESP命令，通知SMMU建立页表结束。
(3)SMMU根据CMD_PRI_RESP生成PRI response返回给PCIE RC。PPR周期结束。

SVA支持

SVA是让CPU和设备使用相同的虚拟地址。SVA就是PCIe上说的SVM（Intel的Shared Virtual Memory）。

例如上图所示，以PCIe RC/EP之间使用DMA搬运数据为例。可以按照如下步骤搬运数据。
(1)在RC申请的地址VA之后，与RC中进程的Pasid一起拿到EP测（PCIe的Pasid就是SMMU所说的SubStreamId）。
(2)在EP测映射iova地址（或者直接使用EP侧的VA地址，如果支持的话），使用DMA触发搬移（当然要将RC的VA地址和EP的iova地址以及EP进程的Pasid写入）。
(3)Device侧DMA在EP侧触发PCIe PRI（Page Request Interface）。EP侧拿到Pasid与VA就可以顺藤摸瓜找到EP侧的PA。之后就可以在EP侧IOMMU建立VA->PA的页表。

要支持SVA机制，需要支持如下特性：
(1)需要支持Pasid或者SubStreamId特性。IOMMU申请Pasid(SubStreamId)然后设备使用其触发访问
(2)IOMMU要支持I/O Page Faults (IOPF)。就是支持接受PCIe PRI或者ARM SMMU stall。
(3)MMU与IOMMU必须要有相同的Page Table格式。

性能

因为SVA访问有Page Fault流程，SVA特性对性能没有太大帮助。SVA特性只是为了简化DMA地址管理。

接口

iommu_sva_bind_device

分配PASID, 注册mmu_notifier()以跟踪所有页表无效化操作，从而保持设备TLB（转换后备缓冲器）的同步。例如，当某个页表条目被标记为无效时，IOMMU（输入输出内存管理单元）会将此无效化操作传递至设备的TLB，这将强制设备未来对此虚拟地址的访问必须通过ATS（地址转换服务）参与地址转换。若IOMMU返回“页面不存在”的响应，设备会在执行I/O操作前，通过PCIe PRI（页请求接口）协议请求将该页面调入内存。
