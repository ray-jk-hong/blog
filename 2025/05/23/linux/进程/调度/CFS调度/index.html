介绍

调度基础篇可以看到CFS的基本的组织形式是在struct cfs_rq中以红黑数的形式将struct task_struct的调度实体struct sched_entity保存，红黑树的key值就是vruntime。

先来看一下几个相关结构体成员的意义:

tasks_timeline: 红黑树根
rb_leftmost: 红黑树中左下角的节点，即下次被调度的任务
min_vruntime: 最小虚拟运行时间，具有此值的task下次将得到运行

load: 权重
run_node: 红黑树中的节点
on_rq: 是否在运行队列中
exec_start: 上次开始执行的时间
sum_exec_runtime: 总共的运行时间
vruntime: 虚拟运行时间

vruntime计算和进程切换

CFS在每次调度的时候，选取红黑树最左下测的调度实体进行调度。保存在红黑树的key值是调度实体的struct sched_entity的vruntime。所以知道vruntime的累加方式至关重要。下面就来看一下vruntime的计算和累加方式。

系统调度周期

所谓系统调用时间是通过__sched_period函数计算的时间。这个时间是CFS认为每个进程可以都执行一遍的时间。

理想的执行时间(ideal_runtime)

通过函数sched_slice函数计算。这个值表示下一次进程最大的执行时间。实际的进程执行时间只能小于等于这个值，因为进程调度流程会保证这个。
理想的执行时间 = （系统调用周期）* (本进程的权值 / 本执行队列的总权值)。
（上述的计算方法可能有些出入，具体要看一下sched_slice函数，后续补充…）

vruntime

vruntime时间 = (实际运行时间) * (nice为0的权值 / 进程的权值)。

总结

  1. nice值
    nice == 0：vruntime=实际运行时间，就是说vruntime增长速度与实际运行时间一致
    nice < 0：vruntime增长速度比实际运行时间慢
    nice > 0：vruntime增长速度比实际运行时间快
    所以进程的优先级越高，vruntime增长越慢，得到调度的机会就越多。

  2. 每个进程每次执行的时间不能大于ideal_runtime

  3. 在一个系统调度周期之后，所有进程的vruntime都相等。后续就重新开始公平调度。
