linus反对c++进入内核的原因

编程语言
|浏览量:

在软件开发的世界中，有一个长期存在的有趣争论，那就是林纳斯·托瓦兹（Linus Torvalds）在Linux内核开发中对C++使用的强烈且一贯的反对态度。作为Linux内核的创始人和维护者，林纳斯·托瓦兹从不掩饰自己的观点，他的主张强调了操作系统开发中需要考虑的一些根本性因素。要理解他的观点，我们需要探讨内核开发的核心原则——性能和稳定性。这两个因素是从智能手机到超级计算机，甚至是关键的医疗设备等系统的基石。托瓦兹认为，引入C++及其相关功能必然会损害这些关键因素。

我们来看看他对C++的尖锐批评是否合理，并逐一分析他的主要技术论点。C语言传统上使用返回值来表示错误，这是一种简单且可预测的方式。当函数出现问题时，它会返回一个特定值，调用者可以检查并适当处理。而C++引入了异常处理机制，允许在代码的任何地方抛出错误。托瓦兹认为，这种方式使得错误处理流程变得不可预测和非确定性，尤其是在像内核这样复杂的环境中，问题更加突出。异常可能会沿着调用栈以意想不到的方式传播和扩散，这使得调试比追踪简单的返回值要困难得多。引入异常可能会导致内核中的错误处理不一致，进而在系统运行过程中引发不稳定性。例如，想象一下MRI扫描仪这样的关键系统，我们真的希望它在意外情况下突然抛出异常吗？

托瓦兹还对C++中隐藏内存分配的语言功能持谨慎态度，特别是C++的RAII设计模式和析构函数等自动内存管理功能。Linux内核使用了高度精细和优化的内存管理方式，而引入自动化的编译器功能可能会导致性能下降，并且使代码依赖于特定编译器的实现方式。这种依赖性在拥有3000万行代码库、运行在数十亿设备上的情况下，绝不是可以轻视的问题。

虽然面向对象编程（OOP）常被认为是C++的主要优势，但托瓦兹认为，基本的OOP概念在C语言中通过结构体已经足够实现。他认为C++中的复杂功能是不必要的负担，无法为内核提供必要的优势，反而会带来不必要的风险和复杂性。面向对象编程的主要优点——模块化提升和代码重用性增强，更多是为了开发者的便利。然而，Linux内核将性能和稳定性置于开发者便利性之上。此外，托瓦兹还担心低效和不必要的抽象化会使维护和重构变得更加困难。如果OOP没有得到谨慎应用，随着时间的推移，代码可能会变得复杂，甚至经验丰富的程序员也可能设计出有问题的类层次结构。

托瓦兹的立场不仅仅是技术上的争论，它有着深刻的哲学基础。他相信，在内核开发这样的底层系统编程中，C语言是最合适的。C语言的清晰表达方式和直接的内存操作功能更符合内核的需求。托瓦兹认为，坚持使用C语言可以吸引那些专注于硬件交互和解决系统根本问题的开发者，并避免他们陷入C++对象模型提供的抽象化中。他还指出，即使开发者在内核中使用C++，最终为了保持效率和可移植性，他们仍可能依赖C风格的编码方式。在这种情况下，C++编译器只会增加不必要的复杂性，而几乎没有提供实质性的好处。

在Linux内核开发这一重要背景下，托瓦兹对C++的反对立场是合理的。这种争论在用户级应用程序中可能显得不那么重要，但在像操作系统内核这样性能和稳定性绝对关键的环境中，他的担忧是合理的。用户级应用程序指的是在操作系统之上运行的软件程序，如网页浏览器、文本处理器和游戏等。与直接与硬件交互不同，用户级应用程序依赖操作系统来管理内存和CPU等资源。因此，由于不直接控制系统，应用程序可以引入更多的抽象和复杂性，而不会对系统稳定性产生重大影响。

托瓦兹的立场强调了开发者便利性与系统根本需求之间的重要平衡。最终，他对C++的坚决反对为我们提出了一个重要的问题：这些功能仅仅是花哨的，还是真正必要的？我们的决定是为了开发者，还是为了用户？这一决定在长期内会产生什么影响？
