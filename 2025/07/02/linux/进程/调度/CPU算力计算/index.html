dmips

  • mips(Million Instructions executed Per Second)：表示CPU每秒执行的指令树（单位是百万条）
  • dmips: d表示dhrystone、dhrystone基准测试是一种用于评估计算机系统中CPU性能的工具。dmips表示cpu在运行这种测试程序时的mips、表示在执行dhrystone测试程序的时候、每秒可以执行几百万条执行令。Linux中用该值表示cpu的算力
  • dmips-mhz: 表示cpu在1MHz的频率下、每秒钟可以执行多少dmips

归一化

数据归一化

数据归一化又叫”特征归一化”（Feature Scaling）或”标准化”。例如将不同单位的数据归一化到统一的值上

归一化方法

最值归一化

将不同的特征归一化到某一个最值上面、Linux中对算力的归一化就是使用这种方式、例如将不同的算力特征归一化到1024。

Z-Score

就是把所有数据归一到均值为0、方差为1的分布中的方式。
公式如下：

：归一化之后的值
：特征的均值
：特征值的标准差
：数据集中的一个特征值

该方法适用于数据分布没有明显的边界，即可能存在极端的数据值，归一化后，数据集中的每一种特征的均值为0，方差为1。
优点：相对于上面的最值归一化，即使原数据集中有极端值，归一化有的数据集，依然满足均值为0方差为1。

CPU频率和算力获取和计算（CONFIG_CPU_FREQ配置并设置DTS场景）

初始化阶段

ARM平台的CPUFreq驱动（如cpufreq-dt、特定SoC驱动）在初始化时会从硬件寄存器或设备树（Device Tree）中读取CPU支持的最大频率。例如：

  • 设备树源：驱动解析设备树中operating-points-v2定义的频率电压表。
  • 硬件寄存器：部分驱动直接读取芯片的ID寄存器获取频率能力。

cpu_scale计算

在如下流程中、根据raw_capacity计算cpu_scale并调用topology_set_cpu_scale()赋值给cpu_scale。

cpu_scale的值通过arch_scale_cpu_capacity()函数获取、其值就是表示cpu能达到的最大算力（归一化到1024、最大就是1024）。

cpu_capacity_orig、cpu_capacity

这两个值都是保存在struct rq中、表示当前调度队列所在的cpu的算力

  • cpu_capacity_orig：表示最大的算力（cpu运行在最大的频率时所能提供的算力）、这个值与上述的cpu_scale相同、值是固定的
  • cpu_capacity：表示剩余算力、该值实际就是cpu_capacity_orig扣除dl/rt/irq消耗的算力之后剩余算力的值、值是变化值

cpu_capacity_orig和cpu_capacity在kernel\sched\fair.c文件的update_cpu_capacity函数中更新。

上述函数中scale_rt_capacity()函数就是在计算剩余算力。所谓剩余算力就是这个cpu的最大算力减去irq/rt/dl任务所消耗的算力之后剩余的算力。
在这里计算剩余算力的原因是为了看能留给cfs的还剩多少算力。为什么对irq的计算要用比例呢？这是为了确保无论多繁忙都要给irq预留一定的空间。（配置了CONFIG_HAVE_SCHED_AVG_IRQ才生效）。

CONFIG_HAVE_SCHED_AVG_IRQ

这个配置需要打开以正确评估中断带来的cpu算力消耗、让其他的调度器能够正确的做迁移。

总结on/off该特性的利弊如下：

特性 不启用时的风险 启用后的改进
中断负载可见性 中断时间完全隐藏 中断时间整合进 CPU 负载
调度决策 基于不完整负载做出错误决策 基于真实负载优化任务放置
频率调节 CPUFreq 低估需求，降频卡顿 按需提频保证中断处理及时性
适用场景 低中断嵌入式设备 服务器/网络/存储等高中断场景
