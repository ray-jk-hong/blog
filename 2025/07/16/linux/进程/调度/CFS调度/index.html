先来看一下几个相关结构体成员的意义:

tasks_timeline: 红黑树根
rb_leftmost: 红黑树中左下角的节点，即下次被调度的任务
min_vruntime: 最小虚拟运行时间，具有此值的task下次将得到运行

sum_exec_runtime: 总共的运行时间
vruntime: 虚拟运行时间

vruntime计算和进程切换

CFS在每次调度的时候，选取红黑树最左下测的调度实体进行调度。保存在红黑树的key值是调度实体的struct sched_entity的vruntime。所以知道vruntime的累加方式至关重要。下面就来看一下vruntime的计算和累加方式。

系统调度周期

所谓系统调用时间是通过__sched_period函数计算的时间。这个时间是CFS认为每个进程可以都执行一遍的时间。

权重（weight）

理想的执行时间(ideal_runtime)

通过函数sched_slice函数计算。这个值表示下一次进程最大的执行时间。实际的进程执行时间只能小于等于这个值，因为进程调度流程会保证这个。
理想的执行时间 = （系统调用周期）* (本进程的权值 / 本执行队列的总权值)。
（上述的计算方法可能有些出入，具体要看一下sched_slice函数，后续补充…）

vruntime

vruntime时间 = (实际运行时间) * (nice为0的权值 / 进程的权值)。

总结

  1. nice值
    nice == 0：vruntime=实际运行时间，就是说vruntime增长速度与实际运行时间一致
    nice < 0：vruntime增长速度比实际运行时间慢
    nice > 0：vruntime增长速度比实际运行时间快
    所以进程的优先级越高，vruntime增长越慢，得到调度的机会就越多。

  2. 每个进程每次执行的时间不能大于ideal_runtime

  3. 在一个系统调度周期之后，所有进程的vruntime都相等。后续就重新开始公平调度。

CFS负载均衡

CFS负载均衡的实现:

  • CFS使用红黑树来维护所有可运行的任务，并根据每个任务的虚拟运行时间进行排序。﻿
  • CFS会选择红黑树中最左边的节点(即虚拟运行时间最小的节点) 作为下一个要执行的任务。﻿
  • CFS还考虑了CPU之间的负载差异，并将新任务分配给相对空闲的CPU 核心。﻿
  • 在一些场景下，CFS还会使用PELT(Per-Entity Load Tracking) 算法来跟踪每个调度实体的负载，从而更精确地进行负载均衡

CPU负载

只有CFS的负载均衡考虑cpu_capacity、RT的负载均衡不考虑。RT的负载均衡有其他的维度。

  • cpu_capacity: 可以用于CFS任务的算力
  • cpu_capacity_orig： 该CPU的原始算力，和微架构及其最大频率相关

负载均衡（load balance）

  1. CFS周期性地检查系统负载，将繁忙CPU 上的任务迁移到空闲CPU 上、以达到负载均衡、这种周期性负载均衡通常在tick 中触发，也称为tick load balance 或periodic load balance。

流程如下：

  1. 当一个CPU 上的任务队列为空，需要进入idle状态时，也会触发负载均衡，将其他CPU 上的任务迁移过来，称为new idle load balance。﻿执行newidle_balance()函数。

流程如下：

  1. 如果一个CPU 负载过高、可以通过IPI (Inter-Processor Interrupt, 处理器间中断) 唤醒其他空闲CPU 进行负载均衡，称为nohz idle load balance。

任务放置（task placement）

  1. 当一个新任务(例如，新创建的线程，或者被唤醒的阻塞线程) 准备就绪时，需要选择一个CPU 核心来执行。这就是任务放置。﻿
  2. FS 会根据CPU 负载情况和任务的特点，选择一个最合适的CPU 核心。﻿
  3. 任务放置主要发生在：(调用select_task_rq函数)
  • 唤醒一个新的线程﻿
  • 执行一个新线程﻿
  • 唤醒一个被阻塞的进程

﻿
4. 在任务放置的过程中，CFS 会考虑CPU 的负载情况，以及任务的亲和性(affinity) 等因素。﻿

主动均衡（active placement）

  1. 当一个低算力CPU 上运行了一个“不合适”的任务(misfit task), 且该任务持续执行，即使负载均衡也无法将其迁移时，就需要主动均衡。
  2. 主动均衡会将这个“不合适”的任务迁移到算力更高的CPU 上，从而实现更好的负载均衡。
