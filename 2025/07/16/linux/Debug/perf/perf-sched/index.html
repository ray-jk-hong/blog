Linux perf 在 Linux 4.10 中获得了一个新的 CPU 调度器分析视图：perf sched timehist。由于我之前没有讨论过 perf sched，我将在这里总结其功能。如果您时间紧迫，浏览以下截图可能会有所帮助，这样您就能了解有哪些可用功能。（我也已将此内容添加到我的 perf 示例页面中。）
perf sched 使用转储和后处理的方法来分析调度器事件，这可能会有问题，因为这些事件可能非常频繁 - 每秒数百万次 - 记录它们会消耗 CPU、内存和磁盘开销。我最近一直在使用 eBPF/bcc（包括 runqlat）编写调度器分析工具，这让我可以通过使用内核内部汇总来大大减少开销。但在某些情况下，您可能希望使用 perf sched 捕获每个事件，尽管开销更高。想象一下，您有五分钟时间分析一个即将自动终止的有问题的云实例，您希望捕获所有内容以便稍后分析。

这是一个 8 CPU 系统，您可以看到从左侧开始的每个 CPU 的 8 列。一些 CPU 列开始时是空白的，因为在配置文件开始时我们尚未跟踪该 CPU 上的事件。它们很快就会被填充。

perf sched timehist

在 Linux 4.10 中添加，按事件显示调度器延迟，包括任务等待被唤醒的时间（等待时间）和唤醒后到运行的调度器延迟（调度延迟）。我们更感兴趣的是调整调度器延迟。
