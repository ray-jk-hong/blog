Linux操作系统，会记录CPU负载，以供调度器进行负载均衡。还会让其他子系统参考这些CPU负载来进行相应调整，例如像DVFS等。

CPU负载分为三种：

  • 全局负载
  • 运行队列负载
  • PELT（Per Entiry Load Tracking）

指数加权移动平均（EWMA）

这里先看一下所谓的指数加权移动平均是什么，因为下面在计算CPU的全局负载/PELT的时候都会用到这种方式来计算。

指数加权移动平均exponentially weighted moving average (EWMA)，又叫指数移动平均exponential moving average (EMA)，是计算平均值的一种方式。

全局负载

全局负载定义

这里所谓的CPU负载，就是每隔一段时间统计CPU的runqueue上是否有TASK_RUNNING和TASK_UNINTERRUPTIBLE类型的进程个数。
例如一个CPU上每隔5秒统计了1分钟，每次查看的时候CPU上都有一个TASK_RUNNING或者TASK_UNINTERRUPTIBLE类型的进程。
那CPU负载统计出来就是1。如果只有一半时间有进程在runqueue上，那统计出来就是0.5，如果每次统计都是2，那统计出来就是2，依次类推。

这里为什么要加上TASK_UNINTERRUPTIBLE类型的？有些人数是TASK_UNINTERRUPTIBLE是等IO操作，所以要加上。这个理由有点牵强~ 后续补充…

调度实体状态

___update_load_sum()函数涉及load/runnable/running这几个状态。这三种状态分别对应struct sched_avg的load_*/runnable_*/util_*成员。
查看__update_load_avg_se()函数可知、对于调度实体来说、load/runnable是一样的、都是!!sched_entity->on_rq来判断。!!sched_entity->on_rq的值是1、表示调度实体已经调用enqueue_entity挂到调度队列中了。值是0表示调度实体已经调用dequeue_entity从调度队列中删除了。
running状态表示、当前进程是否在执行。

accumulate_sum函数中按照如下方式计算并累加时间：
1.delta += sa->period_contrib：上一次记录时间的时候剩下的时间加上经过的时间
2.periods = delta / 1024：计算经过的周期（1ms为周期）
3.sa->load_sum = decay_load(sa->load_sum, periods)：将上次的load_sum乘以衰减系数赋值给当前的load_sum（decay_load算法查看上面的EWMA、y^32 = 0.5方式衰减）
4.contrib = __accumulate_pelt_segments(periods,1024 - sa->period_contrib, delta)：D1乘以n个窗口的衰减（n就是periods）、D2乘以n-1个窗口的衰减、再加上D3得到新的contrib的值
5. 将contrib的值加上去。（直接加或者乘以系数）

RT负载均衡

PUSH and PULL

PULL and PULL都是rt对应的负载均衡方式。

PULL：负载轻的CPU从负载繁重的CPU pull tasks来运行。这应该是主要的方式，因为不应该让负载本身就繁重的CPU执行负载均衡任务。相应的为load balance。
PUSH：负载重的CPU向负载轻的CPU推送tasks由其帮忙执行。相应的为active balance。
总体来讲，PUSH就是排序、把低优先级任务放到合适的队列上面。PULL就是找当前最高优先级进行调度。

https://zhuanlan.zhihu.com/p/662916716

调度器使用push-pull的策略执行任务均衡，主要考虑解决一下4个问题：

  1. 优先唤醒的任务放到哪个运行队列；
  2. 在一个runqueue上，唤醒的任务比当前runqueue上运行的任务优先级低，这个低优先级的任务如何处理；
  3. 在同一个runqueue上，唤醒了一个高优先级任务，它抢占了当前正在运行的低优先级任务，这个低优先级任务如何处理；
  4. 如何处理一个任务主动降低了自己的优先级，导致之前低优先级任务变为了高优先级任务；
    Push-pull策略：
  5. push:策略处理2、3两种情况，从所有的runqueue中找到一个比正在push的任务优先级更低的任务队列；
  6. pull策略处理第4中情况，当前runqueue如果发现当前正在调度的任务优先级比之前的任务低，则尝试从其他runqueue中找到一个更高优先级来运行。
    备注：SCHED_FIFO是主要的调度策略。总体来讲，push就是排序，把低优先级任务放到合适的队列上面，pull就是找当前最高优先级进行调度。

PUSH算法细节：
两种情况触发push动作：

  • 一个低优先级任务被高优先级任务抢占；
  • 一个runqueue上低优先级任务被唤醒，但此时这个runqueue上面正有一个高优先级任务在运行。
    在这两种情况下，为低优先级任务找到合适runqueue的动作称为push动作。push_rt_task()算法首先从当前runqueue中找到一个没有运行的优先级最高的任务A，然后从所有的
    cpu中找到一个比A优先级低的最低优先级cpu集合(因为优先级低，所有可以被抢占)。那么如何
    从这几个cpu中找到一个最合适的cpu呢？基于以下原则：
  1. 如果只有一个cpu，最简单，就是它了；
  2. 其次看cpu集合是否包含了A上次使用的cpu(可能基于cache一致性考虑)；
  3. 再次考虑逻辑上和A上次运行的cpu最接近的cpu；
  4. 如果上面的都不成功，则从cpu集合中随机找到一个；

PULL算法细节：
以下情况触发pull动作：

  • 一个任务即将恢复执行:假设进程A即将在cpu 1上恢复执行， pull_rt_task()算法会遍历系统中所有的runqueue，判断是否能找到优先级比A高的没有运行的进程。如果能找到，则把这些进程pull到cpu 1上运行。

备注：这个pull动作可能会迁移不止一个任务到cpu 1上面，这个是基于以下考虑，如果只
迁移一个任务到cpu 1 上面（非原子操作），真正运行的时候，可能发现这个任务已经在其
他cpu上运行了，因为其他cpu都在并行运行。所以可能会迁移多个任务，但这又可能造成
在未来的某个时间引起push操作，造成乒乓效应，现实中这种情况很少出现。

root_domain and sched_domain

cpu优先级管理

  • cpu的优先级等于这个cpu上面实时进程的最高优先级；
  • 可以处于一下几种状态： INVALID, IDLE, NORMAL, RT1, … RT99；
  • 当cpu处于INVALIDs时，不能参与调度

pri_to_cpu: 根据优先级找寻对应的cpu
pri_activ：此domain具有哪些优先级
cpu_to_pr：根据cpu找他的优先级

cpupri是root_domain结构中的一员，用来维护cpu->进程优先级和进程优先级->cpu的对应关系

备注：如果一个cpu的优先级是50，那么pri_active[50+2] = 1。因为pri_active[0]和pri_active[1]是留给INVALID, IDLE两个状态的
