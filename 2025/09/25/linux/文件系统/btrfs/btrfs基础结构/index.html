介绍

假设你要在btrfs中查找并读取某个文件中的内容，看一下btrfs是怎么一步一步处理请求的。

超级块

和所有文件系统一样，你第一个能读到的而且也是最容易读到的就是本文件系统的超级块，因为超级块本身在磁盘中的大小和偏移都是固定的。
btrfs的superblock在磁盘的64KB偏移的地方，这个定义在#define BTRFS_SUPER_INFO_OFFSET (64 * 1024)。

btrfs对应的超级块定义是btrfs_super_block，要在读到超级块之后解析后续要访问的树的根。所以btrfs_super_block首先要保存树根的逻辑块地址还要保存一个所谓的sys_chunk_array数组来讲树根的逻辑块地址转换成物理块地址。主要的几个成员如下：

映射树（chunk_root）

chunk_root保存了所有逻辑块到物理块的转换信息。chunk_root本身的读取，需要依赖btrfs_super_block的sys_chunk_array，其逻辑块地址也是保存在btrfs_super_block->chunk_root中。
上一节中，文件系统挂载过程将btrfs_super_block->sys_chunk_array解析并保存在了struct btrfs_chunk_map。
此后就可以通过btrfs_super_block->chunk_root以及btrfs_map_block函数就可以拿到chunk_root对应的物理块并完成chunk_root读取了。

树根(root tree)

在chunk_root读取完成之后、我们就可以读取其他树的树根了。树根的逻辑地址是保存在btrfs_super_block->root中。

此种类型的key.type为BTRFS_ROOT_ITEM_KEY。

root_tree本身是一种树、此树中保存了很多其他类型树的根。btrfs中保存的树有以下集中：

  • device allocation tree
  • extent allocation tree
  • fs tree
  • checksum tree
  • log root tree

上述的类型是通过key.objectid来判断的、例如fs tree的objectid就是BTRFS_FS_TREE_OBJECTID。所以例如我们要找fs tree相关的结构，可以通过判断
(key.type==BTRFS_ROOT_ITEM_KEY && key.objectid==BTRFS_FS_TREE_OBJECTID)来判断并获取到fs tree相关的struct btrfs_root_item。

文件系统树(fs tree)

文件系统树包含三种类型的项（item）：

  • DIR_ITEM：实现了目录结构。它将文件名映射到对应的inode号码。一个目录查找就是一层一层查找对应的DIR_ITEM的过程
  • INODE_ITEM：包含了文件的元数据，类似于 ext4 的 inode。例如：文件权限 (mode)、所有者 (uid, gid)、文件大小、创建/修改/访问时间 (atime, mtime, ctime)。注意：它不直接包含指向数据块的指针。
  • EXTENT_DATA_ITEM：这是最关键的项，它真正将文件内容与磁盘上的数据连接起来。它记录了文件内的逻辑偏移量(offset)、这个偏移量对应的数据在磁盘上的物理位置(一个extent、就是一段连续的物理块)、对于小的文件EXTENT_DATA_ITEM还可以将数据直接保存在里边避免寻找物理块。

在查找文件的过程中、就是查找DIR_ITEM的过程。在上一节的流程中、已经找到fs tree对应的树的根也就是btrfs_root_item。btrfs_root_item中的bytenr就是表示fs tree的树根的位置。后续的fs tree的数据的组织方式和上述的B-tree的查找方式都是一样的，只不过payload会转成btrfs_dir_item而已。
例如我们要查找/home/user/data.log文件、btrfs_root_item就是顶级目录、也就是/目录。那在/目录下查找home目录的方式如下：

到这里只是查找到了/home这个目录。现在这里暂停一下看一下btrfs_dir_item中都有哪些信息：

  • localtion： 一个 btrfs_key 结构，它包含了这个目录条目指向的那个文件或目录的 inode 号码。这是连接目录和文件的“指针”
  • transid： 创建此目录项的事务 ID，用于快照等操作
  • data_len: 紧随文件名之后的额外数据的长度。对于普通的目录项，这个值通常是 0。
  • name_len: 这是最关键的字段。它是一个 16 位的无符号整数，明确记录了文件名的长度（单位是字节）。最大长度是 65535，但受限于 Btrfs 块大小和 POSIX 规范（通常为 255 字节）。
  • type: 文件类型，如 BTRFS_FT_REG_FILE (普通文件), BTRFS_FT_DIR (目录), BTRFS_FT_SYMLINK (符号链接) 等。

btrs_dir_item是个可变结构体、紧随type后保存着文件名。name_len就是保存着文件名的正确的大小。例如btrfs_dir_item表示的文件名是”test.txt”、那name_len就是8、没有在后面添加’\0’。

接下来我们再看一下btrfs文件系统是怎么逐级解析路径/home/user/data.log的、并写入数据的.

查找

在上一步找到的文件系统树中，进行 B-Tree 查找操作。

  1. 查找home: 使用一个btrfs_key（由父目录的inode号和文件名home构成）进行查找，找到home目录对应的DIR_ITEM。获取home的inode: 从DIR_ITEM中获得home目录的inode号。
  2. 查找user: 使用一个新的btrfs_key（由home的inode号和文件名user构成）进行查找，找到user目录对应的DIR_ITEM。获取user的inode: 从DIR_ITEM中获得 user 目录的inode号。
  3. 查找data.log: 使用btrfs_key（由user的inode号和文件名data.log构成）进行查找，找到data.log文件对应的DIR_ITEM。获取data.log的 inode: 从DIR_ITEM中获得data.log文件的inode号。
  4. 上一步知道了data.log的inode号、在本文件系统树中查找INODE_ITEM、即btrfs_inode_item结构体。这样我们就可以知道data.log的文件大小/修改时间等等信息。

写入

如果要写入数据到data.log文件中、要进一步按如下步骤进行写操作：(此时写时复制(COW)此时生效)

分配新的物理空间 (Extent)

需要为写入data.log的数据找到新的磁盘块、此时查找”分配树 (Extent Allocation Tree)”找到大小合适的空闲Extent、它会以COW方式更新树、将这块Extent标记为已使用。

将数据和校验和写入磁盘

数据写入过程有以下几个步骤：

  1. 在上面分配的Extent磁盘块中写入数据
  2. 计算这个新数据块的 CRC32C 校验和
  3. 将计算出的校验和（以 CoW 方式）写入到 校验和树 (Checksum Tree) 中

创建新的元数据并触发 CoW 级联更新

上面写磁盘和更新的都是数据内容和校验内容。完成之后就要更新元数据了、这是最核心的一步。由于旧的元数据不能被修改，所有相关的元数据项和B-Tree节点都必须被复制和更新。

  1. 创建 EXTENT_DATA_ITEM: 创建一个新的 btrfs_extent_data_item。这个 item 将 data.log 文件内的逻辑偏移量（即原文件末尾的位置）映射到第 6 步中新数据写入的物理位置。

  2. 更新 INODE_ITEM: 创建一个 btrfs_inode_item 的副本，更新其中的 size（文件大小）和 mtime（修改时间）等字段。

  3. CoW 级联更新:

    • 将新的extent_data_item和更新后的inode_item插入到 文件系统树 中
    • 这会导致包含它们的叶子节点被复制一份，然后修改。
    • 指向这个叶子节点的父节点也必须被复制和修改，以指向新的叶子节点。
    • 这个“复制-修改”的过程会一直向上传播，直到 文件系统树 的根节点。最终，我们得到一个全新的、指向新世界的文件系统树根节点。

提交事务

更新上层树

现在我们有了一个新的文件系统树根，需要让顶级的btrfs_root_item知道。
更新过程是：在btrfs_root_item中，找到指向/home/user/data.log所在子卷的那个ROOT_ITEM，通过 CoW 方式更新它，使其指向上一部分写入过程中创建的那个全新的文件系统树根。这个过程同样会产生一个全新的根树根节点。

提交事务，更新超级块

这是最后一步，让整个文件系统的变更“生效”。
修改内存中的超级块，将其中的root指针更新为”更新上层数”这一步产生的新根树的物理地址。
将更新后的 超级块 写入到磁盘的固定位置。为了安全，btrfs会写入多个副本。
一旦超级块成功写入磁盘，整个写操作就原子性地完成了。现在文件系统的入口指向了包含所有新数据的全新B-Tree结构。所有旧的树节点和数据块都保持不变，它们将在没有快照引用的情况下，在未来被清理回收。
